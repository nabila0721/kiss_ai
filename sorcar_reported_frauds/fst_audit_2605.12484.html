Bottom line

The paper is a real, technically coherent piece of work from a senior author list (Joseph E. Gonzalez, Matei Zaharia, Kurt Keutzer, Inderjit Dhillon, Rishabh Agarwal), not a fabricated submission. All referenced arXiv IDs resolve, the cited companies (Eragon, Periodic Labs, Mirendil) exist, the named reflection LM gpt-5.2 is a real OpenAI model (visible via OpenRouter/LiteLLM catalogs), and the video and blog links work. We found no evidence of fabricated identities, fabricated datasets, or fabricated references.

However, the paper has serious reproducibility and presentation problems that materially affect how its headline numbers should be read. Most importantly: the “Code” link advertised on the first page is a placeholder page (“Code coming soon”), the central “FST has lower KL than RL” figure silently drops the one task where the effect does not hold, and the headline “3× more sample-efficient” claim is matched-step, not matched-compute, while the appendix itself notes that an end-to-end FST run is more expensive than RL.

2. Critical reproducibility issue: code not released

The PDF advertises three live artifacts on its title page:

Page 1 hyperlinks (extracted from PDF):
  Video → https://rishabhtiwari.ai/projects/fst/video.mp4   (200, 3.7 MB)
  Blog  → https://rishabhtiwari.ai/projects/fst/blog/        (301 → staging URL, see §5)
  Code  → https://rishabhtiwari.ai/projects/fst/code/        (200, but content is a stub)

The Code URL returns this page (full HTML retrieved by curl):

“Code coming soon. We’re cleaning up the codebase before the public release. Check back shortly.”

There is also no companion GitHub repository: GitHub search for fast-slow-training LLM returns zero hits, and the only repos by the named GitHub accounts (github.com/devvrit and the GEPA org gepa-ai/gepa) do not contain an FST training loop. The principal empirical claims of the paper — the 3× sample-efficiency curves, the 70% KL reduction, the plasticity collapse-to-zero on RL, and the continual-learning HoVer→CodeIO→Physics trajectory — are therefore not reproducible by anyone outside the author group at this time. This is not, by itself, fraud (many ML papers release code months later), but it is a hard reproducibility blocker.

3. Cherry-picking by panel omission (Figure 4)

The main-text Figure 4 is the central evidence for “FST shifts the accuracy/KL frontier further left.” It shows three panels: CodeIO, HoVer, Physics. Polaris (Math) — one of the three headline training families — is absent. The paper does not mention the omission anywhere in the main text or captions.

The Polaris KL panel only surfaces in Appendix G, where the authors admit:

“The Polaris (math) trajectories in Figure 11 look qualitatively different from the other three tasks: RL and FST sit on top of each other in (KL, reward)-space rather than FST pulling the frontier to the left. … The three-task variant in the main text drops the Polaris panel for the reasons discussed below.”

Mathematically the “reason” is the Polaris run uses a non-public custom-SFT base. Operationally the consequence is that the headline 4-task KL narrative is supported by only 3 of the 4 tasks, and the reader of the main paper is not told. This is a textbook case of cherry-picking by panel exclusion — the underlying data is preserved in the appendix, but the figure that everyone looks at hides the contradicting evidence.

4. Headline efficiency claim is matched-step, not matched-compute

The abstract and Figure 2 advertise:

“FST is up to 3× more sample-efficient than only slow learning (RL).”

Readers will interpret this as “3× cheaper to train.” That is not what the data show. From Appendix D (verbatim):

“Mean per-RL-step wall-clock under the headline configuration is ~60s for RL-only and ~100s for FST without rollout reuse (HoVer-hard, K=8).”
“This figure covers the RL training loop only: FST additionally runs periodic GEPA cycles (rollouts for K candidate prompts plus a reflection call), which add real wall-clock on top, so end-to-end a FST run is more expensive than an RL-only run of equal step count. … GEPA reflection calls to gpt-5.2 are billed separately; at the per-cycle metric-call budget above this is ≤ $10 per training run.”

So the headline “3× fewer training steps” collides with an admitted ≥1.67× per-step cost (without rollout reuse) plus extra GEPA cycles plus paid GPT-5.2 calls. The actual matched-compute speedup is unstated and is not 3×. The matched-compute number is the one that matters to a practitioner, and it is conspicuously absent from the main paper. With “rollout reuse” (Appendix F) the per-step gap narrows but does not close end-to-end.

5. Blog hosted on a staging URL

The “Blog” link on the title page 301-redirects to:

https://blogpost-rl-gepa.gepa-docs-staging.pages.dev/blog/2026/05/11/learning-fast-and-slow/

That destination is the staging deployment of the GEPA documentation site (the URL slug literally contains “gepa-docs-staging”). The blog’s JSON-LD also lists the publisher as “GEPA”. Hosting an arXiv-paper blog on a co-author’s product staging environment is unconventional and undermines the impression of an independent evaluation — see §6.

Five of the nine FST authors are also authors of the very methods FST is built out of and benchmarked against. The natural baseline “GEPA only” (Figure 8) and “GEPA augmenting RL” (Figure 2 radar) are implementations the same group’s own tool ships. The blog is hosted on the GEPA team’s documentation infrastructure (§5). None of this is fraud — it is disclosed in the author list — but it does mean that “FST beats RL” and “FST beats GEPA-only” should be read as internal consistency checks of the GEPA + ScaleRL stack, not as independent validations.

7. Sigmoid-extrapolation asymptote

Figure 3’s headline numbers (CodeIO +4.4pp, Math +2.9pp, HoVer-hard +7.7pp) come from fitting a 4-parameter sigmoid ΔR = (A−R0)/(1+(Cmid/C)B) and reading off A, the upper asymptote. The paper explicitly states:

“Solid curves cover the fit window; dotted curves are extrapolation past the last training step.”

Reading off the asymptote of a saturating curve fit when the run has been stopped before saturation is statistically unstable: A trades off with B and Cmid and is most poorly constrained exactly when the run was stopped early. No fit-uncertainty interval is reported for A. The +2.9pp HoVer-style claim in particular depends heavily on how far the dotted extrapolation is pushed. This is a methodologically weak way to report a “ceiling” difference and would be flagged in any rigorous peer review.

8. No seeds, no error bars

We grepped the entire PDF for “seeds”, “n=3”, “standard error”, “variance”, “confidence interval” in the context of training runs. No multi-seed protocol is described anywhere. The radar charts, the asymptote fits, the plasticity drop-to-zero, and the continual-learning trajectory are all consistent with single-seed runs per (task, method). Combined with the early-stopping protocol (“RL is trained until step 1500 or in-distribution saturation, whichever comes first; FST is trained at least until it matches RL’s running peak”), this gives the authors substantial researcher degrees of freedom. The “collapses to ~0% within 40 steps” plasticity claim on Math→HoVer-hard (Figure 5) is exactly the kind of extreme result that needs replication across seeds before being headlined.

9. Custom “HoVer-hard” and custom Polaris base

  • HoVer is a public benchmark (Jiang et al. 2020, arXiv:2011.03088), but the paper uses a subset labelled “HoVer-hard.” The subset-selection rule is not given. With code not released (§2), an external party cannot regenerate the same hard split.
  • The Math (Polaris) RL run uses a custom base model: “Qwen3-8B base further SFT’d on Nemotron.” Neither the SFT recipe nor the checkpoint is released. This base is what produces the conveniently-different KL behaviour that lets Figure 4 drop the Polaris panel (§3).

10. Concurrent work substantially under-credited: E-SPL

Reference [64] is “Evolutionary System Prompt Learning for Reinforcement Learning in LLMs” (Zhang, Chen, Stadie, arXiv:2602.14697, submitted Feb 2026, ~3 months before FST). Its abstract describes:

“… in each RL iteration, E-SPL samples trajectories under multiple system prompts in parallel, then jointly applies RL updates to LLM weights and evolutionary updates to system prompts. System prompts evolve via mutation and crossover … selection is based on relative performance ratings updated across RL iterations. E-SPL encourages a natural division between declarative knowledge encoded in prompts and procedural knowledge encoded in weights ….”

That description is operationally indistinguishable from FST’s “fast textual weights and slow parametric weights co-evolve” framing — right down to the declarative/procedural split that FST claims in its Discussion. FST’s entire Related Work treatment of E-SPL is one sentence: “Recent work E-SPL [64] explores prompt optimization and RL at a smaller scale with focus on performance rather than adaptation.” That is a meaningful under-attribution. It is not citation fraud (the reference is present), but the central novelty positioning of FST is weaker than the main text suggests.

11. Other smaller issues

  • Citation year mismatch. Reference [2] (GEPA, arXiv:2507.19457) was submitted 25 Jul 2025 but the FST bibliography lists it as “2026.” The PDF on arXiv has not been updated to a 2026 version. Minor sloppiness, not fraud.
  • Equation 7 formatting. Main text Figure 3 caption gives the sigmoid as R−R0 = (A−R0)/(1+(Cmid/C)B), but Eq. (7) is written as ΔR = (A−R0)/… with no definition of ΔR. Inconsistent notation, minor.
  • Star-graph “0/50 pass@16” is explicitly seeded on a single prompt (“pass@16 is 0/50 on the seed prompt before any RL update”). The dramatic “FST escapes zero by step 50” claim is contingent on that one prompt. The seed prompt is included in Appendix C (good), but the “escape” effect should be robust across reseeds; no such ablation is reported.
  • “1-bit-per-episode information limit” (page 8) is asserted without citation and overstates the RLVR setting; binary reward + group-relative advantages carry more than 1 bit per group, and the actual signal-to-noise depends on G, not strictly 1 bit per episode. Rhetorical, not a numerical fraud.

12. Security vulnerabilities — not applicable

There is nothing to attack here in the cryptographic / web sense: no executable code is published, no server endpoints are exposed for adversarial probing, and the only network-reachable artifacts are static (a 3.7 MB MP4, a stub HTML page, and a markdown-rendered blog). The reflection LM is a third-party API (gpt-5.2 via LiteLLM) outside the paper’s threat surface. The nearest thing to an “attack” one could mount is training-time prompt injection of the GEPA reflection LM via adversarial trajectories, since the reflection prompt embeds the rollout text verbatim — but that is a known feature of any reflection-based optimizer, not a vulnerability of FST specifically, and we cannot construct a real POC without the training harness (see §2).

13. AI-slop check

We did not detect AI-slop hallmarks. The paper has a clear thesis, real ablations, unusual-but-coherent design choices (the K sweep, the “Problem vs Prompt advantage baseline” ablation, the rollout-reuse mechanism), and prose written by humans with idiosyncratic voice (e.g., the “1-bit-per-episode information limit” line). The numerical entries in Tables 1–2 are internally consistent. The cited references all exist. The figures have radar-chart axes labelled with real benchmarks (HMMT25, LCB-Hard, Physics from sciknoweval). This is a real research paper, with the specific problems enumerated above.
